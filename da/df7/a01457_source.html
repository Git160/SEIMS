1 SEIMS开发规范
2 ---------------------
3 
4 + 开发规范的制定是为了保持模型代码的一致性和可读性。
5 + 开发规法包括代码风格统一和模块编写指南
6 
7 ## 1.代码风格规范
8 
9 ### 1.1.代码整理格式规范
10 
11 SEIMS模型主要编程语言为C/C++和Python，采用CMake组织代码。由于不同开发者所用的编辑器各异，因此我们基于Google代码格式规定了C/C++和Python的代码书写格式规范，位于`./doc/style`文件夹下。这两个文件分别可以导入CLion和PyCharm软件，使用菜单栏命令“Code”->“Reformation Code”便可以一键整理代码。
12 
13 ### 1.2.禁用Tab键
14 不同IDE对Tab键的显示不同，Visual Studio下排版漂亮的代码在其他IDE下很可能错位，因此**强烈建议**在VS中将**Tab键用4个空格代替**。
15 
16 打开Visual Studio，设置“工具”->“选项”->“文本编辑器”->“C/C++”->“制表符”->“缩进”选择“智能”，“制表符大小”和“缩进大小”均为4，选中“插入空格”。
17 
18 ### 1.3.不混用换行符
19 众所周知，`Windows`系统和`Linux/Unix`系统的换行符不同，而`Windows`下的Git默认情况下会在提交代码时自动替换换行符`LF`或`CR`为`CRLF`，这样对跨平台代码造成了极大的不便。
20 
21 因此，我们**强烈建议**代码采用`Linux`编码`LF`即`\n`，因此需要做到：
22 + VS或其他IDE设置换行符为LF而非CRLF
23 + 文本编辑器建议采用Notepad++，**禁止**使用Windows的记事本！
24 
25 打开Git shell进行如下设置：
26 
27 ```
28 # AutoCRLF, 设置提交、检出时均不转换
29 git config --global core.autocrlf false
30 # SafeCRLF, 设置拒绝提交包含混合换行符的文件，并且在提交混合换行符的文件时给出警告，从而手动转换为LF换行符后提交
31 git config --global core.safecrlf true
32 git config --global core.safecrlf warn
33 ```
34 
35 详见[Wiki-Github使用说明](https://github.com/lreis2415/SEIMS2017/wiki/Git-guidance)。
36 
37 ### 1.4.行宽
38 行宽必须限制！！！
39 
40 建议100字符为限，太长则应换行。
41 
42 PyCharm中的设置为：File->Settings->Editor->Code Style->Default Opetions->Right margin (columns): 100
43 
44 随后使用Code->Reformat Code (Ctrl+Alt+L)时便会自动换行了。当然自动的并不一定是最好的，还是需要自己手动调整。
45 
46 ### 1.5.括弧
47 在复杂的条件表达式中，用括弧清楚地表示逻辑优先级。
48 
49 ### 1.6.断行与空白的花括号{}
50 避免使用精简风格，如：
51 ```cpp
52 if (condition) DoSomething();
53 else DoSomethingElse();
54 ```
55 虽然这样可以节省几行空间，但是对于单步调试查看各个变量的变化情况非常不便，因此推荐采用以下两种风格：
56 ```cpp
57 if (condition) {
58  DoSomething();
59 }
60 else {
61  DoSomethingElse();
62 }
63 ```
64 或者
65 ```cpp
66 if (condition)
67 {
68  DoSomething();
69 }
70 else
71 {
72  DoSomethingElse();
73 }
74 ```
75 
76 ### 1.7.分行
77 不要把多条语句放在一行！！！更严格地，不要把多个变量定义在一行，以下三行都是不可取的例子。
78 
79 ```cpp
80 int a =1, b = 2;
81 c = 3; d = 5.f;
82 if (fFoo) Bar();
83 ```
84 
85 ### 1.8.命名大小写
86 采用一个通用的做法：所有的类型、类、函数名都用Pascal形式（即所有单词的第一个字母都大写），所有的变量都用Camel形式（即第一个单词全部小写，随后单词随Pascal形式，也称lowerCamel）。
87 
88 + 类、类型、变量采用名词或组合名词，如Member、productInfo等；
89 + 函数则用动词或动宾组合词来表示，如get/set、RenderPage()等。
90 
91 ### 1.9.注释规范
92 + SEIMS采用Doxygen代码注释规范，在Visual Studio中配合VA Assist插件可高效插入注释，[查看详细配置](https://github.com/lreis2415/SEIMS/wiki/Develop-environment#%E5%BC%80%E5%8F%91%E8%BE%85%E5%8A%A9%E5%B7%A5%E5%85%B7-vassistx%E4%BB%8B%E7%BB%8D)。**以下为补充Tips**。
93 + `#ifndef ... #endif`宏命令之后要注释上这是对应的哪个宏命令，以防嵌套使用时混淆不清，尤其是中间代码过长的时候，如：
94  ```cpp
95  #ifdef windows
96  #define Tag_ModuleDirectoryName "\\"
97  #define SEP "\\"
98  #define Tag_DyLib ".dll"
99  #else
100  #define Tag_ModuleDirectoryName "/"
101  #define SEP "/"
102  #define Tag_So "lib"
103  #endif /* windows */
104  #ifdef linux
105  #define Tag_DyLib ".so"
106  #elif (defined macos) || (defined macosold)
107  #define Tag_DyLib ".dylib"
108  #endif /* linux */
109  ```
110 + SEIMS中Python代码注释采用Google风格的docstring，在PyCharm中设置为：
111  `File --> Settings --> Tools --> Python Integrated Tools --> Google`
112 ## 2.代码设计规范
113 建议阅读：[Google开源项目风格指南（中文版）](http://zh-google-styleguide.readthedocs.io/en/latest/contents/)
114 ### 2.1.Python
115 + 运行pylint检查代码中潜在bug或警告，并改正。默认地，pylint会产生很多警告信息，有些是我们不希望看到的，可通过`--disable=`禁止，比如：
116  + 变量、函数、类命名不规范，invalid-name (C0103)
117  + 函数参数过多（多于5个），虽然参数多了风格不好，但是对于变参数的函数设计来讲，参数多于5个很常见，too-many-arguments (R0913)
118  + 某个函数有太多局部变量，too-many-locals (R0914)
119  + 没必要的括弧，如为了兼容python3在print后加的括弧，superfluous-parens (C0325)
120  ```bash
121  pylint C:\z_code\Hydro\SEIMS\seims\preprocess --files-output=y --disable=C0103,R0913,R0914,C0325
122  ```
123 + 导入包或模块时，应使用模块的全路径，而非相对路径
124  ```python
125  # Spatial delineation
126  from seims.preprocess.sd_delineation import SpatialDelineation
127  ```
128 + 命名约定
129  `module_name, package_name, ClassName, method_name, ExceptionName, function_name, GLOBAL_VAR_NAME, instance_var_name, function_parameter_name, local_var_name.`

133 ### 3.1.各分支命名规范及功能定义
134 
135 |分支|命名示例|功能|
136 |:---:|:---:|:---:|
137 |`master`|`master`|稳定版本发布，非工作区，仅管理员可操作|
138 |`dev`|`dev`|开发汇总，非工作区，开发组成员均可将完成的工作合并至此，当`dev`足够稳定之后，合并至`master`|
139 |`bug-<bugcode>-<author>`|`bug-ompconfused-zlj`|bug修复，临时工作区，修复并测试完成之后，将此分支合并至`dev`，并新建`issue`说明问题之后，删除该分支|
140 |`update-<module>-<author>`|`update-docs-zlj`|模块更新，常驻工作区，完成后合并至`dev`、新建`issue`、删除该分支|
141 |`new-<module>-<author>`|`new-lisemsed-zlj`|新增模块，常驻工作区，完成后合并至`dev`、新建`issue`、删除该分支|

147 ### 3.2.分支管理一般步骤
148 
149 + 3.2.1. [加入SEIMS开发组](https://github.com/orgs/lreis2415/teams/watershed_modeling)，直接`clone`实验室账号下的SEIMS库（git@github.com:lreis2415/SEIMS.git）即可，不用`fork`到自己账号下，建议仅克隆`dev`分支，因为默认情况下会克隆所有分支，不仅速度慢，而且浪费空间且没必要，操作如下：
150  ```shell
151  git clone git@github.com:crazyzlj/SEIMS.git --branch dev --single-branch
152  ```
153 + 3.2.2. 发现代码中出现了bug、某个文档需要修改、某个模块需要更新、亦或是需要增加一个新的模块等，新建一个分支，修复它，合并至dev分支，并新建issue告知大家，也是备忘，完后删除这个分支，流程如下：
154 
155  ```
156  # 假设目前cd到SEIMS所在文件夹了
157  git checkout -b bug-IFoundU-zlj
158  # 修改完成，提交修改
159  git add .
160  git commit -m "bug fixed of IFoundU"
161  # 切换回dev分支，并合并刚才的修改
162  git checkout dev
163  git merge --no-ff -m "merge from bug-IFoundU-zlj" bug-IFoundU-zlj
164  # 如果出现冲突，处理之后，可以删除bug分支了
165  git branch -d bug-IFoundU-zlj
166  # 去提交一个issue告知大家吧！https://github.com/lreis2415/SEIMS/issues
167  ```
168 + 3.2.3.删除远程分支。如果某个分支工作已经完成，且已经合并至`dev`分支，在本地可以直接通过`git branch -d tmpbranch`命令删除，远程分支则可推送一个空分支到远程分支，实现删除`git push origin :tmpbranch`
